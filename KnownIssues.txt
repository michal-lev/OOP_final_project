Known Issues / Bugs:

Doctor's hashCode Dependency on Articles

Description: In the Doctor class, the equals and hashCode methods use the articles Set.
Because of this, if someone tries to add or remove articles after a Doctor is already placed inside a HashSet
(like the lecturers Set in the College class), the Doctor's hashCode may change.

This is a known design limitation. In the current system, articles are not modified after creation,
so the issue does not affect runtime behavior.

Description:
The system was refactored into a layered architecture improving separation of concerns and maintainability.

- CollegeUI introduced to handle all user interaction and menu flow

2. Backup System Integration

Description:
Persistence logic was separated into a dedicated service.

IMPORTANT:
Previously, save/load functionality was implemented inside College class.
It was moved into BackupService.

- BackupService handles serialization/deserialization
- College no longer contains file handling logic
- Main handles backup loading and saving

Files & Functions:

BackupService.java:
- loadCollege()
- saveCollege()

Main.java:
- initializeCollege()

College.java (REMOVED):
- save/load related methods

3. Name Handling Refactor (SOLID Improvement)

Description:
Name handling logic was split to separate UI from validation:

- isUniqueName(...) remains in College (business validation only)
- getUniqueName(...) moved to CollegeUI (user interaction + loop handling)

This separation enforces that College remains free of user interaction logic while CollegeUI handles input repetition and user guidance.

5. Moving Input Responsibility from Business Logic to UI Layer

Description:
Article input handling was moved from College to CollegeUI.

- Previously College handled article collection logic
- Now CollegeUI is responsible for collecting article data during lecturer creation
- College receives prepared Set<String> as parameter

This ensures that College only receives fully constructed data structures and does not depend on user interaction logic.

College.java:
- createNewLecturer(...)
- getArticlesNames(...) (removed)

CollegeUI.java:
- addLecturer() (handles article input)

---------------------------------
6. System Entry Point Refactor (Main → CollegeUI)

Description:
The Main class was refactored into a minimal bootstrap component.

A major architectural change was the relocation of the entire menu system:

- The full menu logic (previously implemented in Main using switch-case) was moved to CollegeUI
- Main no longer handles any user interaction or application logic
- Main is now responsible only for:
  - Loading the College instance (from backup if available)
  - Creating a new College if needed
  - Starting the CollegeUI loop

This change enforces a strict separation between application entry point and presentation layer responsibilities.

Files:

Main.java:
- main()
- initializeCollege()

CollegeUI.java:
- startMenu()
- all menu handling methods
